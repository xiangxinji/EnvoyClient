## Context

Member task execution currently runs through a pipeline that produces a plan, executor summary, reviewer summary, attempts, and aggregated `AgentStep` traces. The task result can submit the trace back to Manager as a task resource, and the execution panel can show the live process. The system also initializes a per-user brains directory at `~/.envoy/brains/{username}/` with `raw/日志/`, `raw/偏好/`, and `skills/`, and the existing brains sync flow can sync changed brains files after task completion.

The missing piece is a controlled bridge between completed task execution and durable Agent memory. The Agent should be able to preserve reusable lessons, but it should not automatically rewrite its own skills based on one task.

## Goals / Non-Goals

**Goals:**
- Generate a structured post-task reflection after AI task execution finishes.
- Persist reflections as append-only markdown files in the current user's brains.
- Include enough context to understand what happened without storing excessive raw trace output.
- Clearly mark failed-task reflections so they are not confused with successful patterns.
- Add a user setting that controls whether automatic reflection memory is enabled.
- Reuse existing file and brains sync infrastructure where practical.

**Non-Goals:**
- Do not automatically edit `skills/*.md`.
- Do not build vector search or semantic retrieval in this change.
- Do not add a new Manager API for reflection storage.
- Do not change the task dispatch/result REST contract unless existing result data is insufficient.
- Do not make reflection generation block task resolution indefinitely.

## Decisions

### D1: Add reflection as a post-pipeline stage

**Decision:** Add reflection after the existing planner/executor/reviewer pipeline completes and before `resolveCurrentTask()` returns the final task result.

**Rationale:** The reflection stage needs the complete task context: original task, plan, execution summary, review result, attempts, success/failure state, and trace. Running it after review keeps task execution and knowledge capture separate.

**Alternatives considered:**
- Let the executor write memory directly using `file_write`: rejected because execution and long-term memory mutation would be coupled, and a failed executor could still write authoritative-looking knowledge.
- Add reflection inside reviewer: rejected because reviewer is responsible for pass/fail judgment, not memory curation.

### D2: Store daily execution logs first

**Decision:** Reflection output SHALL be appended to a daily markdown file under `~/.envoy/brains/{username}/raw/日志/执行日志/YYYY-MM-DD.md`.

**Rationale:** A daily execution log is easier for humans and future Agents to scan than one file per task. It preserves task-level entries while keeping the knowledge base compact and date-organized. It also fits the existing brains directory model and sync flow.

**Alternatives considered:**
- Maintain a single `agent-lessons.md`: rejected for initial implementation because merging and deduplication introduce higher risk.
- Create one file per task: rejected because it creates too many small files and makes daily review harder.
- Write directly into `skills/`: rejected because skills directly influence future Agent behavior and need stronger promotion rules.

### D3: Use a concise fixed Chinese log schema

**Decision:** Each daily log file starts with `# 执行日志`. Each task entry uses this fixed structure:
- `## 任务标题：<short generated title>`
- `### 任务描述`
- `### 执行过程`
- `### 任务结果`
- `### 执行复盘`

**Rationale:** The knowledge base should contain refined task information, not full trace dumps. A predictable concise schema makes the log easy to read and easy for future retrieval or summarization logic to parse.

### D4: Generate a concise task title

**Decision:** The task title SHALL be generated by summary logic from task content, execution summary, and review output. It should be short, specific, and sanitized.

**Rationale:** Raw task content is often too long or conversational to work as a daily log heading. A generated title improves scanning while retaining the original task description in the body.

### D5: Use a separate log evaluator agent

**Decision:** Add a dedicated log evaluator agent that receives task content, execution outputs, review output, and trace summary, then returns a structured log draft containing title, description, process, result, and review fields.

**Rationale:** Planner, executor, and reviewer outputs often contain markdown headings, JSON payloads, and detailed trace fragments. A separate evaluator keeps log-writing concerns out of execution and can normalize these noisy sources into concise human-readable daily log entries.

**Alternatives considered:**
- Reuse reviewer output directly: rejected because reviewer output may be JSON and audit-oriented rather than log-oriented.
- Only use deterministic string cleanup: useful as fallback, but insufficient for producing consistently readable summaries.

### D6: Gate reflection with member settings

**Decision:** Add a member setting, defaulting to enabled, that controls automatic task reflection memory.

**Rationale:** The product behavior is to learn from every Agent task by default, while still giving users explicit control to disable the behavior. Reflection writes user-local knowledge and may sync to cloud if brains sync is enabled.

**Alternatives considered:**
- Always-on reflection without a user setting: rejected because task content may contain sensitive data or one-off details.
- Manual-only reflection: safer, but does not satisfy automatic post-task learning.

### D7: Reflection should be best-effort

**Decision:** Reflection failure SHALL NOT fail an otherwise completed task. The task result should still resolve, while reflection errors are surfaced in execution monitoring or console diagnostics.

**Rationale:** Knowledge capture is secondary to task completion. A file write failure, model error, or malformed reflection should not block task flow or leave the ClientTask queue stuck.

### D8: Sanitize and limit stored content

**Decision:** Reflection SHALL summarize trace data instead of dumping full raw tool output, and SHOULD omit obvious secrets, tokens, passwords, private keys, and very large command/file outputs.

**Rationale:** Trace data can contain sensitive local paths, command output, or credentials. Reflections should capture reusable reasoning, not every observed byte.

## Risks / Trade-offs

- **[Knowledge pollution]** → Store reflections as logs first and mark promotion candidates instead of mutating skills automatically.
- **[Sensitive data leakage]** → Add sanitization and content-length limits before writing markdown.
- **[Unbounded growth]** → Use one file per day with concise task entries.
- **[Concurrent writes]** → Member task execution is queue-driven and normally serial. If parallel local execution is added later, introduce an append-specific Tauri command or write lock.
- **[Reflection latency]** → Treat reflection as best-effort and enforce timeout handling if model generation is used.
- **[Bad failed-task memories]** → Mark failed reflections explicitly and separate "mistakes" from "useful lessons".
- **[Sync side effects]** → Reuse existing user-controlled brains sync settings; reflection storage itself stays local file based.
- **[Evaluator output malformed]** → Parse structured output defensively and fall back to deterministic cleanup when the evaluator returns invalid data.

## Migration Plan

- Add the reflection log template directory to `src-tauri/brains/raw/日志/执行日志/` with `.gitkeep`.
- Add the new member setting with a safe default for existing users.
- Existing brains directories are updated by the current incremental `init_brains` behavior when users log in after upgrade.
- Existing tasks and task resources do not need migration.

## Open Questions

- Should automatic reflection default to off or on for new users?
- Should log evaluator output later include typed memory candidates, or should daily logs remain purely human-readable?
- Should users get a UI for browsing reflections in this change, or should browsing remain via the existing brains browser?
