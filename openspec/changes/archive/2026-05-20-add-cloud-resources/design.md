## Context

当前项目已有两套文件存储机制：
1. **任务资源**（`~/.envoy/teams/{team}/tasks/{id}/resources/`）— Agent 上传的任务产出物，生命周期绑定任务
2. **聊天附件**（`~/.envoy/attachments/{team}/{date}/`）— 聊天中发送的文件，生命周期绑定消息

两者都不适合作为团队共享文件空间。任务资源随任务结束而归档，聊天附件散落在对话中无法统一浏览。团队需要一个独立的、持久化的共享文件仓库。

Manager 使用 Hono + better-sqlite3，每个团队有独立的 `team.db`。客户端使用 Vue 3 + Tauri，通过 REST API 与 Manager 通信。Agent 通过 `useTaskExecution` 注册工具，调用 Manager AI 推理端点进行 ReAct 循环。

## Goals / Non-Goals

**Goals:**
- 提供团队级嵌套目录文件仓库，支持浏览、上传、下载、创建目录、删除操作
- 通过 SQLite `cloud_files` 表追踪文件元信息（上传者、大小、时间）
- 权限控制：所有成员可浏览/上传/创建目录，仅 Leader 可删除
- Agent 可通过工具访问和上传云资源
- Manager 管理后台展示团队云资源使用统计

**Non-Goals:**
- 文件版本控制 / 历史记录
- 文件大小限制和配额管理（后续迭代）
- 文件搜索功能
- 文件分享链接 / 外部访问
- 在线预览 / 编辑

## Decisions

### D1: 存储架构 — 文件系统 + SQLite 元数据

**选择**: 实际文件存储在 `~/.envoy/teams/{teamName}/cloud/` 的真实目录结构中，元信息通过 `cloud_files` SQLite 表追踪。

**备选方案**:
- A) 纯文件系统 + `.meta.json` 侧车文件（与项目 JSON 文件存储风格一致）
- B) 文件系统 + SQLite（当前选择）
- C) 纯数据库存储（BLOB）

**理由**: 方案 B 在并发安全、查询灵活性（按用户统计、按路径搜索）上优于 A，在文件大小处理上优于 C。SQLite 已在项目中使用（better-sqlite3），无新依赖。

### D2: 目录浏览 — 单层级列表 + 面包屑导航

**选择**: API 返回指定路径下的直接子项（一层），客户端通过面包屑导航逐级浏览。

**备选方案**:
- A) 单层列表（当前选择）
- B) 递归返回完整目录树

**理由**: 单层列表 API 更简单，避免大目录树的性能问题。客户端可以按需加载，用户体验更可控。

### D3: 删除策略 — Leader 级联删除

**选择**: 删除目录时级联删除所有子文件和子目录（磁盘 + DB 记录），仅 Leader 有权限。

**理由**: 简单直接。团队规模有限，级联删除不会造成性能问题。通过二次确认弹窗防止误操作。

### D4: 路径安全 — 规范化 + 白名单

**选择**: 所有用户提供的路径在服务端进行 `path.resolve` + `normalize` 处理，验证结果路径以 cloud 根目录为前缀，防止路径遍历攻击。

**理由**: 文件系统操作必须防止 `../../etc/passwd` 类攻击。规范化路径后做前缀检查是最可靠的方式。

### D5: API 路由组织 — 独立路由文件

**选择**: 新建 `manager/server/routes/cloud.ts`，在 `index.ts` 中注册，遵循现有路由文件模式。

**理由**: 与 `messages.ts`、`teams.ts` 等现有路由文件保持一致的组织方式。云资源是独立的功能域，不混入现有路由。

### D6: Agent 工具 — cloud_list + cloud_upload

**选择**: 新增两个 Agent 工具，通过 Manager REST API 操作。

**工具定义**:
- `cloud_list(path)`: 列出指定目录内容 → `GET /api/cloud/files?path=`
- `cloud_upload(path, filename, content)`: 上传文件到指定路径 → `POST /api/cloud/files`

**理由**: Agent 在 ReAct 循环中需要读取团队共享文件和上传执行结果到共享空间。走 Manager API 而非直接操作文件系统，保持统一的权限和元数据追踪。

## Risks / Trade-offs

**[路径遍历攻击]** → 所有路径参数经过 `resolve` + `normalize`，验证以 cloud 根目录为前缀。文件名额外过滤 `..` 和特殊字符。

**[并发上传同名文件覆盖]** → 上传时检查同名文件是否已存在，存在则拒绝（返回 409），客户端提示用户。不做版本控制以保持简单。

**[大目录删除阻塞]** → 目录删除是同步的文件系统操作。对于超大目录可能耗时较长。当前阶段团队规模有限，可接受。后续如需优化可改为异步 + 进度追踪。

**[磁盘空间无限制]** → 当前不设配额，依赖运维监控。Manager 后台统计面板可帮助发现异常使用。
